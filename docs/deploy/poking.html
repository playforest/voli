Poking around the server

A cookbook of curl and shell commands for verifying that voli serve is up, exploring its endpoints, and chasing down "why isn't this working" moments.

Everything here is provider-agnostic. Replace $VOLI_URL and $VOLI_TOKEN with your local or deployed values and the commands work the same.

Setup variables#

Pick the right pair of values for the server you're poking:

# Local dev with --no-auth
export VOLI_URL=http://localhost:8080
export VOLI_TOKEN=             # empty; --no-auth servers ignore it

# Local dev with auth on
export VOLI_URL=http://localhost:8080
export VOLI_TOKEN=$(echo $VOLI_AUTH_TOKEN)   # whatever you set when starting

# Phase B: Docker container running locally
export VOLI_URL=http://localhost:8080
export VOLI_TOKEN=$(grep ^VOLI_AUTH_TOKEN .env.deploy | cut -d= -f2)

# Phase C: deployed on DigitalOcean
export VOLI_URL=https://voli.example.com
export VOLI_TOKEN=$(grep ^VOLI_AUTH_TOKEN .env.deploy | cut -d= -f2)

For a server with auth on, every protected request needs:

-H "Authorization: Bearer $VOLI_TOKEN"

Health and OpenAPI are different. /healthz is always public. /openapi.json is protected, so a public deploy doesn't leak the tool catalogue to crawlers.

Liveness check#

curl -s $VOLI_URL/healthz
# {"status":"ok"}

Anything other than HTTP 200 here means the process isn't running, the port is wrong, or a reverse proxy is in front and misconfigured. Add -v to see the request and response headers when you need to debug.

Inspect the OpenAPI spec#

The full spec describing every REST tool:

curl -s $VOLI_URL/openapi.json -H "Authorization: Bearer $VOLI_TOKEN" | jq .

Quick "what tools exist":

curl -s $VOLI_URL/openapi.json -H "Authorization: Bearer $VOLI_TOKEN" \
  | jq -r '.paths | keys[] | sub("^/tools/"; "")'
# get_underlying_snapshot
# list_option_contracts
# get_option_quotes
# get_option_greeks
# compute_atm_iv_term_structure
# compute_skew_slope
# get_atm_greeks

The exact request schema for one tool:

curl -s $VOLI_URL/openapi.json -H "Authorization: Bearer $VOLI_TOKEN" \
  | jq '.paths["/tools/get_underlying_snapshot"].post.requestBody.content."application/json".schema'

Call individual tools via REST#

This is exactly how ChatGPT Custom GPT Actions invoke voli. You can reproduce any GPT-driven call from the shell.

Spot snapshot

curl -s -X POST $VOLI_URL/tools/get_underlying_snapshot \
  -H "Authorization: Bearer $VOLI_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"ticker": "NVDA"}' | jq .

List option contracts

curl -s -X POST $VOLI_URL/tools/list_option_contracts \
  -H "Authorization: Bearer $VOLI_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"ticker": "NVDA", "expiry": "2026-05-16", "right": "C", "limit": 20}' \
  | jq '.contracts | length'

Get quotes for specific symbols

curl -s -X POST $VOLI_URL/tools/get_option_quotes \
  -H "Authorization: Bearer $VOLI_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"option_symbols": ["O:NVDA260516C00200000"]}' | jq .

Get greeks (vendor-published only)

curl -s -X POST $VOLI_URL/tools/get_option_greeks \
  -H "Authorization: Bearer $VOLI_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"option_symbols": ["O:NVDA260516C00200000"], "mode": "vendor_only"}' \
  | jq .

Analytics shortcuts (one call each)

curl -s -X POST $VOLI_URL/tools/compute_atm_iv_term_structure \

MCP over curl#

Claude.ai's MCP transport over HTTP is JSON-RPC 2.0 wrapped in SSE. Real clients (claude.ai, Claude Desktop, the MCP Inspector) handle the session handshake automatically; for spot-checks, you can drive it by hand. The minimum useful flow:

# 1. Initialize: opens a session, returns a session ID in the
#    `mcp-session-id` response header.
SESSION=$(curl -s -i -X POST $VOLI_URL/mcp \
  -H "Authorization: Bearer $VOLI_TOKEN" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json, text/event-stream" \
  -d '{
        "jsonrpc": "2.0",
        "id": 1,
        "method": "initialize",
        "params": {
          "protocolVersion": "2025-06-18",
          "capabilities": {},
          "clientInfo": {"name": "curl-poke", "version": "0.0.1"}
        }
      }' | awk -F': ' '/^mcp-session-id/{print $2}' | tr -d '\r\n')
echo "session: $SESSION"

# 2. List tools.
curl -s -X POST $VOLI_URL/mcp \
  -H "Authorization: Bearer $VOLI_TOKEN" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json, text/event-stream" \
  -H "mcp-session-id: $SESSION" \
  -d '{
        "jsonrpc": "2.0",
        "id": 2,
        "method": "tools/list",
        "params": {}
      }'

If you need a friendlier MCP poking experience, use the MCP Inspector. Point it at $VOLI_URL/mcp with the bearer token; it shows tools, arguments, and call results in a UI.

Auth scenarios#

Verify the auth middleware is doing its job.

# No token: 401
curl -s -o /dev/null -w "%{http_code}\n" \
  $VOLI_URL/tools/get_underlying_snapshot \
  -X POST -H "Content-Type: application/json" -d '{"ticker":"NVDA"}'
# 401

# Wrong token: 401 (server gives the same generic message; can't tell)
curl -s -o /dev/null -w "%{http_code}\n" \
  $VOLI_URL/tools/get_underlying_snapshot \
  -X POST -H "Content-Type: application/json" \
  -H "Authorization: Bearer wrong-token" -d '{"ticker":"NVDA"}'
# 401

# Wrong scheme (Basic instead of Bearer): 401
curl -s -o /dev/null -w "%{http_code}\n" \
  $VOLI_URL/tools/get_underlying_snapshot \
  -X POST -H "Content-Type: application/json" \
  -H "Authorization: Basic $(echo -n :$VOLI_TOKEN | base64)" \
  -d '{"ticker":"NVDA"}'
# 401

# Right token: 200
curl -s -o /dev/null -w "%{http_code}\n" \
  $VOLI_URL/tools/get_underlying_snapshot \
  -X POST -H "Content-Type: application/json" \
  -H "Authorization: Bearer $VOLI_TOKEN" -d '{"ticker":"NVDA"}'
# 200

# Public path (always works)
curl -s -o /dev/null -w "%{http_code}\n" $VOLI_URL/healthz
# 200

What each error code means#

CodeMeaningLikely cause
200Tool ran, response is in the bodyworking
400Malformed JSON, or body is not an objectcheck the -d payload
401Missing / wrong / non-Bearer auth headercheck $VOLI_TOKEN matches $VOLI_AUTH_TOKEN on the server
404Unknown tool name in the URLcurl ... /tools/<typo>; cross-check with /openapi.json
406(/mcp only) client didn't send Accept: text/event-streamonly matters for MCP requests; REST tools don't need this
500Tool ran but raised; body has {"error", "message"}usually means Polygon failed: no key, bad ticker, rate limit
502 / 504Reverse proxy in front of voli is timing out or can't reach itCaddy / Cloudflare / DO health-check misconfiguration

Watching the server#

In one terminal, run with verbose logging:

poetry run voli serve --port 8080 --no-auth --server-url http://localhost:8080 \
  2>&1 | tee /tmp/voli-serve.log

In another, hit the endpoints from this doc and watch the log scroll in real time. Useful for confirming requests actually reach the server when you suspect a TLS, proxy, or port issue.

To see Polygon HTTP traffic specifically (e.g. to confirm the cache is serving repeats):

POLYGON_HTTP_DEBUG=1 poetry run voli serve --port 8080 --no-auth

Every Polygon request prints a one-liner to stderr; cache hits are silent.

Phase B: Docker variations#

Once the Dockerfile lands in Phase B, all the commands above work unchanged against the containerised server:

# Build, run with --no-auth (local dev)
docker build -t voli-server .
docker run --rm -it --env-file .env.deploy -p 8080:8080 voli-server \
  voli serve --port 8080 --server-url http://localhost:8080

# Then in another terminal:
curl -s http://localhost:8080/healthz

Or with the bearer token enforced:

docker run --rm -it -e VOLI_AUTH_TOKEN=$(python -c 'import secrets; print(secrets.token_urlsafe(32))') \
  -e POLYGON_API_KEY=$POLYGON_API_KEY \
  -p 8080:8080 voli-server \
  voli serve --port 8080

Container logs:

docker logs -f <container-id>

Phase C: deployed variations#

Once the server is on a public URL behind HTTPS, the commands stay the same; only $VOLI_URL changes. Useful first checks:

# Server is reachable
curl -sS $VOLI_URL/healthz

# TLS cert is valid (no -k anywhere; if you need -k, the cert is wrong)
curl -sSI $VOLI_URL/healthz | head -1

# OpenAPI spec advertises the right `servers` URL (must match $VOLI_URL)
curl -s $VOLI_URL/openapi.json -H "Authorization: Bearer $VOLI_TOKEN" \
  | jq '.servers'

If /openapi.json returns a servers URL that doesn't match the host ChatGPT is calling, ChatGPT will fail to call the tools. The server URL is set with --server-url when you start voli serve; rebuild or restart with the right value.
